优化 SpecSync 扩展的四个改进方向

## 当前状态
SpecSync 是一个 VS Code 扩展，用于检测 Spec 文档和代码之间的不一致性。当前版本已实现基础功能。

## 需要优化的四个方向

### 1. 从注释依赖升级到 AST 分析
当前规则检测依赖注释关键词（如验证、TODO）。
改进目标：
- 使用 TypeScript Compiler API 解析代码 AST
- 检测实际的验证逻辑（如 if 语句、正则表达式）
- 识别字段验证、长度检查、格式验证等模式

### 2. 从命名推断升级到装饰器标记
当前 API 匹配基于文件名和函数名推断。
改进目标：
- 支持 @Spec() 装饰器标记 API 实现
- 支持 @Validate() 装饰器标记验证逻辑
- 支持 @Rule() 装饰器标记业务规则
- 保持向后兼容（命名推断作为 fallback）

### 3. 扩展多语言支持
当前只支持 TypeScript。
改进目标：
- 添加 Python 支持（解析 docstring、类型注解）
- 添加 Java/Kotlin 支持（解析 Javadoc、注解）
- 设计 LanguageParser 接口，便于后续扩展
- 根据文件扩展名自动选择解析器

### 4. 深度逻辑验证（可选增强）
当前只检查有没有，不检查对不对。
改进目标：
- 验证数值范围（如长度 6-20 是否真的有 if 判断）
- 验证正则表达式（如邮箱格式、密码复杂度）
- 验证枚举值（如 status 是否只返回 pending/paid/shipped）

## 技术要求
- 保持现有测试通过
- 新功能需要单元测试
- 代码需要编译通过（npm run compile）
- 保持与现有架构风格一致

## 实现顺序建议
1. 先实现 AST 分析（影响最大）
2. 再实现装饰器支持
3. 然后添加 Python 支持作为多语言示例
4. 最后实现深度逻辑验证

完成后运行测试确保没有破坏现有功能。
